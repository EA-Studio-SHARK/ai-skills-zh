# prompt-zh

用于中文提示词工程、Agent 指令、任务拆解、文案改写和结构化输出。目标是让 AI 的输入输出更像真实中文团队协作，而不是翻译腔、营销腔或空泛 AI 腔。

## 核心规则

- 先写目标，再写约束，最后写验收；因为 Agent 需要明确优先级。
- 使用具体动词，如“修改”“新增”“删除”“验证”；因为抽象词会导致执行边界模糊。
- 避免“赋能”“深度探索”“不容忽视”等套话；因为这些词信息密度低且 AI 腔明显。
- 明确是否允许提问；因为有些任务需要直接执行，有些任务需要先澄清。
- 指定工作目录或文件范围；因为 Agent 可能在错误位置操作。
- 写清不能破坏的现有行为；因为 AI 容易为简化实现删掉业务逻辑。
- 验收标准要可检查；因为“效果更好”无法验证。
- 输出格式要给示例；因为中文表格、列表、JSON 的期望容易不一致。
- 复杂任务要拆成阶段；因为 Agent 更容易跟踪进度和回滚。
- 不要把多个无关任务塞进一个 prompt；因为上下文冲突会降低执行质量。
- 写清目标受众；因为面向老板、开发者、用户的表达不同。
- 指定语气时用具体描述；因为“高级感”“专业”太宽泛。
- 要求引用资料时说明来源范围；因为 AI 可能编造来源。
- 要求代码时说明技术栈和版本；因为不同版本 API 差异很大。
- 要求中文输出时说明是否保留英文术语；因为技术文常需要中英混排。
- 禁止模型自夸或解释自身能力；因为这类内容对用户任务无帮助。
- 让 AI 说明假设；因为未确认前提会影响结论可靠性。
- 高风险领域要求保守表达；因为医疗、法律、金融不能给绝对承诺。
- 对长文要求结构和字数范围；因为无约束长文容易散。
- 对代码任务要求运行测试或说明无法测试原因；因为交付需要验证。
- 对改写任务要求保留事实，不新增未提供信息；因为润色不应编造。
- 对翻译任务要求意译还是直译；因为中文自然度和准确性取舍不同。
- 对内容创作要求给反例；因为反例能约束模型不要踩常见坑。
- 对 Agent 指令避免礼貌废话；因为系统指令越短越稳定。

## 反例

```md
请你深度赋能我的项目，全面优化用户体验，并输出一个专业高级的方案。

问题：目标、范围、验收都不清楚。
```

```md
帮我写一个非常好的公众号文章，要有爆款感，越详细越好。

问题：没有受众、主题、字数、结构和禁忌，容易生成模板文。
```

## 正例

```md
你是一个前端开发 Agent。请直接修改当前仓库的用户管理页面。

目标：
新增“批量导出用户”功能。

约束：
- 使用现有 Ant Design 组件。
- 不改接口返回结构。
- 导出按钮只在已选择用户时可点击。
- 导出失败要展示中文错误原因。

验收：
- 选择 1 个或多个用户后可以导出 CSV。
- 未选择用户时按钮禁用。
- `npm test` 通过；如果无法运行，说明具体原因。
```

```md
把下面这段话改成自然中文，保留事实，不要新增案例，不要使用“赋能、闭环、抓手、底层逻辑”。

输出：
1. 改写版
2. 修改说明，最多 3 条
```

## 配置建议

- 通用 Agent prompt 使用固定结构：角色、目标、上下文、约束、输出、验收。
- 内容 prompt 使用固定结构：受众、场景、核心观点、语气、禁用词、示例。
- 代码 prompt 默认写明：直接修改、保持现有功能、运行验证、列出变更文件。
