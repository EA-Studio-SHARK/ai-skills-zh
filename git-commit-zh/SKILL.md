# git-commit-zh

用于生成、评审和规范中文 Git commit message。基于 Conventional Commits，同时适配中文团队的可读性、问题追踪、PR 审查和 changelog 生成。

## 核心规则

- 使用格式 `<type>(<scope>): <中文摘要>`；因为机器可解析且人类易读。
- 摘要使用中文动宾结构；因为“修复登录失败”比“登录失败修复”更自然。
- 摘要控制在 50 个中文字符内；因为 git log 和 GitHub 列表空间有限。
- `type` 使用英文固定枚举；因为生态工具能识别 conventional commits。
- 常用 `type` 包含 `feat`、`fix`、`docs`、`style`、`refactor`、`perf`、`test`、`build`、`ci`、`chore`、`revert`；因为这些覆盖多数变更。
- `scope` 使用模块名或业务域；因为读者需要快速定位影响范围。
- 不确定 scope 时可以省略；因为错误 scope 比无 scope 更误导。
- 一个 commit 只表达一个主题；因为混合提交难以回滚和 review。
- 正文说明“为什么”和“怎么改”；因为摘要只能说明改了什么。
- 修 bug 时正文写触发条件；因为后续排查需要复现信息。
- 破坏性变更使用 `!` 或 `BREAKING CHANGE`；因为发布和升级需要明确提醒。
- 不要使用 `update`、`fix bug`、`修改代码` 这类空泛摘要；因为无法判断变更内容。
- 不要在摘要末尾加句号；因为 commit 标题通常不是完整段落。
- 中文和英文术语之间保留空格；因为中英混排更易读。
- issue 或需求编号放正文或 footer；因为标题里堆编号会影响阅读。
- 自动格式化单独提交；因为格式化会掩盖真实逻辑 diff。
- 依赖升级要写版本和原因；因为升级可能引入兼容风险。
- 回滚提交使用 `revert:` 并说明回滚目标；因为需要追踪被回滚 commit。
- WIP 提交只用于临时分支，不进主干；因为主干历史应可读。
- 合并提交信息要保留 PR 语义；因为 squash 后只剩一条记录。
- 测试变更用 `test`，不是 `fix`；因为测试本身不一定改变产品行为。
- 性能优化用 `perf` 并写指标或目标；因为“优化”很容易空泛。
- 重构用 `refactor`，前提是不改变外部行为；因为改变行为应使用 `feat` 或 `fix`。
- 文档更新用 `docs`，不要混入代码修复；因为文档变更审查重点不同。

## 反例

```text
update
```

问题：没有类型、范围和内容。

```text
fix: 修复问题
```

问题：不知道修复了什么问题，也不知道影响模块。

```text
feat: 大幅度优化系统，增强用户体验，完善多个模块并修复若干 bug
```

问题：一个提交混合多个主题，无法回滚和审查。

## 正例

```text
feat(payment): 支持支付宝退款单查询

新增退款单号查询接口，并把支付宝返回状态映射为内部退款状态。
用于客服后台排查退款处理中和退款失败的订单。
```

```text
fix(miniprogram): 避免商品列表重复加载同一页

onReachBottom 在低端机上可能连续触发，导致同一页商品重复追加。
本次新增 loading 和 finished 判断，保证同一时间只有一个分页请求。
```

```text
perf(article): 减少公众号预览页图片首屏加载

- 首屏外图片改为懒加载
- 封面图增加固定宽高，避免布局抖动
- CDN 缓存时间调整为 7 天
```

```text
refactor(user): 拆分用户筛选表单组件

不改变查询参数和接口调用，仅将筛选区拆成独立组件，便于后续复用。
```

## 配置建议

- 推荐 commitlint 配置保留英文 `type`，摘要允许中文。
- 团队可约定 scope 枚举，如 `payment`、`miniprogram`、`admin`、`article`、`prompt`。
- 生成 commit 前先读取 staged diff，不要根据未暂存文件编写提交。
- PR squash 时使用最能代表整体变更的 commit message。
